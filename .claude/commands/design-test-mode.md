# /design-test-mode — 测试用例设计模式

基于 design.md 设计测试用例。**不写代码，不给解决方案，不受现有实现影响。**

---

## 启动前提：design.md 完备性检查

启动时必须先执行以下检查，**任一不通过则拒绝进入模式并告知原因**：

1. **唯一性**：在当前分支对应的实验目录（`~/data/test/` 或 `~/data/exp/` 下与当前 git branch 相关的目录）中，找到且仅找到一个 `design.md`
2. **存在性**：文件确实存在，且非空
3. **完整性**：内容不包含以下任何不完整标志——
   - `TODO`、`TBD`、`待完成`、`待实现`、`待讨论`、`未完成`、`暂定`
   - 只有标题没有内容的空节（`## 标题\n\n##` 这种结构）
   - `[...]`、`...（待补充）` 等占位符

检查通过后，读取 design.md 全文，告知用户"已读取 design.md，准备开始测试用例讨论"，并简述 design.md 涉及的核心接口和功能范围。

---

## 能力边界

**可以：**
- 阅读 design.md、design-require-changes.md 和相关源文件
- 在对话中讨论测试用例、边界条件、接口联动
- 向用户追问需求细节（须通过提问门槛，见下）
- 将最终确认的测试用例写入 `test_case.md`（与 design.md 同目录）

**不可以：**
- 编写任何实现代码或测试代码（只写测试用例描述，不写 Python/unittest 代码）
- 提供解决方案或实现建议
- 受现有代码实现的影响——测试用例只服务于 design.md 的设计方案和用户需求，不是对现有实现的复述

---

## 工作流程

### 第一步：读取所有已有材料（提问前必须完成）

按顺序执行，不可跳过：

1. 读取 design.md 全文
2. 若同目录存在 `design-require-changes.md`，完整读取——其中所有 Q&A 视为**已知事实**，不得重问
3. 对 design.md 中引用的接口，读对应源文件确认已有实现语义（如 DataSource 如何命名 factor、SparseData 的字段结构等）

### 第二步：需求确认（严格过滤）

完成第一步后，对每一个候选问题，必须通过以下两个 gate，**两个都通过才能提问**：

**Gate 1 — 不可自行推导**：能从 design.md、design-require-changes.md、源码中直接推导出答案的，**不问**。

**Gate 2 — 影响测试内容**：如果答案是 A 而非 B，至少一个测试用例的断言内容会因此不同。如果不影响任何断言，**不问**。

通过两个 gate 的问题，**一次性列出，上限 3 个**。

### 第三步：测试用例讨论

围绕以下四类展开，逐类讨论并确认：

**① 单元测试（单接口，隔离测试）**
- 正常路径：典型输入 → 预期输出
- 边界值：最小值、最大值、空输入、单元素、最大容量
- 异常路径：非法类型、超界值、违反约束的输入 → 预期异常类型

**② 接口联动测试（多接口协作）**
- 典型使用序列（按 design.md 中描述的调用顺序）
- 状态依赖：某接口的输出作为另一个接口的输入
- 顺序敏感性：调换调用顺序是否应该失败？

**③ 不变式验证**
- design.md 中标注的约束（"不修改 self.x"、"返回值非空"等）是否可以通过测试验证？
- 对象内部状态在操作前后的一致性

**④ 回归边界**
- 哪些场景是"容易写错实现"的？这些场景必须有对应测试用例
- 哪些是 design.md 明确排除的行为？（有时候"不应该发生"也需要一个测试来确认它确实不会发生）

每个测试用例讨论完后，**验证覆盖完备性**：
> "以上用例是否覆盖了 [接口X] 所有可能的输入情况？有没有遗漏的分支？"

### 第四步：写入 test_case.md

所有测试用例经用户确认后，写入与 design.md 同目录的 `test_case.md`，格式如下：

```markdown
# Test Cases

> 基于：[design.md 路径]
> 版本：[当前日期]

## [接口/功能名]

### [测试用例名称]
**类型**：单元 / 联动 / 不变式 / 回归
**前置条件**：（初始状态描述）
**输入**：（具体值或类型说明）
**预期输出/行为**：（精确描述，包括异常类型）
**覆盖的需求点**：（对应 design.md 中的哪条约束或接口）
```

每个测试用例独立成节，不写实现代码，只写行为描述。

---

## 退出模式

用户说"开始写测试代码"/"动手实现测试"时，退出此模式，切换为正常实现流程（此时 test_case.md 作为测试实现的规格参考）。
