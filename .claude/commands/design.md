# /design — 创建或更新设计文档

当用户需要记录、讨论或恢复某个功能的设计方案时调用此 skill。

---

## 文件位置与分支绑定

设计文档与实验目录绑定，不随 git 管理：

| 类型 | 路径 | 说明 |
|------|------|------|
| 临时验证 | `/home/peiran/data/test/<exp-dir>/design.md` | 可随实验产物一起清除 |
| 持久化设计 | `/home/peiran/data/exp/<exp-dir>/design.md` | 不可随意删除 |

`<exp-dir>` 通常以功能分支命名（如 `multi-stock-pool-generator`），在 `description.md` 的 **分支** 字段中记录当前 git branch，保证实验目录与代码版本可对应。

---

## 何时创建 / 更新

- 功能涉及**新接口、跨模块交互、架构决策**时，在开始写代码前建立 design.md
- 讨论中出现方案选择或重要约束时，即时更新，不积压到事后整理
- 重复运行实验时保留 design.md（只清理实验产物），如设计有变则更新对应部分

---

## 文档结构

### 第一部分：接口规格

每个接口只写四件事：

1. **签名**（参数名 + 类型 + 默认值）
2. **参数/返回值表**（每项一行，写约束而非行为描述）
3. **关键约束**（不直观的限制，一句话，比如"不修改 self.x"、"返回 True 表示找到至少一个"）
4. **交互说明**（若该接口涉及多个对象的协作，一句话说明调用顺序或依赖）

不写算法细节，不写实现步骤，不写示例代码（除非签名本身无法表达行为）。

已实现的接口用状态表格列出（位置 + 一行说明），不重复展开。

### 第二部分：设计说明

按**决策主题**组织，不按类/方法组织。每条说明格式：

> **[选择]**：选了 A 而非 B，因为 C。

没有被否定备选方案的说明不算设计说明，只是描述。

---

## 写作原则

- **先写接口，再写理由**：接口是事实，理由是解释，分开写让读者能快速定位
- **约束比行为更重要**：行为从命名可以猜，约束是踩坑的地方
- **不写过程，只写结果**：设计文档不是讨论记录，只保留最终决策；中间的讨论过程留在对话里
- **跨 session 可恢复**：文档应能让下一个 session 在不读代码的情况下理解当前设计状态，并直接开始实现
